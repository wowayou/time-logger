## 0. 一句话

不要把本阶段理解为「继续实现一份产品路线图」。

时间尺已经具备可日常使用的记录闭环。当前唯一被个人证据支持的价值是：

> **以尽可能低的管理成本留下可信行动记录，让自己第二天仍能知道昨天做过什么。**

本阶段的核心不是证明它「值得成为一个大产品」，而是让它**停止吞噬现实目标**：先稳定地用它记录生活，同时把求职投递和作品集真正交付出去。

## 1. V1 定义（本阶段硬标准）

- 每日记录和管理成本中位数 ≤ 5 分钟；
- 第二天能依据时间线重建昨天的主要活动；
- 数据不发生不可解释的丢失或静默损坏；
- 完整备份可在隔离的空环境中恢复出可核对的数据；
- **记录本身有价值，不要求 AI 复盘必须改变行为。**

### 价值层级

| 层级 | 当前状态 |
|---|---|
| 重建真实一天 | 已由个人长期 Dogfooding 初步证明 |
| 理解时间去了哪里 | 当前产品可支持，继续观察 |
| 识别目标偏差 | 候选增值层 |
| 改变第二天行动 | 待验证，**不作为 V1 门槛** |
| 内建 AI / Agent 集成 | 冻结，不实施 |

### 产品与作品集的关系

第一优先级是维护者自己愿意持续使用；第二优先级是自然沉淀为可展示的工程作品。**作品集不是新增功能的理由**——案例必须基于已完成的工作和真实工程取舍整理。

## 3. 权威设备与时区协议

**iPhone SE 是唯一权威记录设备。**

| 语义 | 本轮取值 |
|---|---|
| 采集设备时区 | `Asia/Tokyo` / GMT+9 |
| 原始记录时间 | iPhone 写入的本地时间字符串 |
| 实际生活分析时区 | `Asia/Shanghai` / GMT+8 |
| 自然日归属 | 分析结论用 GMT+8；iPhone 界面仍按设备本地日展示 |

iPhone 上的原始记录、原始 JSON 备份和原始时间字符串是**权威事实，不得改写**。不要求为本轮实验修改 iPhone 系统时区。

东京 00:00–00:59 对应 GMT+8 前一天的 23:00–23:59；分析副本转换必须正确处理跨日归属。（Tokyo 与 Shanghai 均不实行夏令时，本观察窗内偏移恒为 −60 分钟，不会漂移。）

### GMT+8 分析副本规则

只有在需要按真实生活日分析时才创建派生副本：

1. 先从 iPhone 导出不可变的原始完整备份；
2. 原始文件保留，不覆盖、不编辑；
3. 在隔离的浏览器配置、独立 origin 或离线环境中创建副本；
4. 使用明确的 `Asia/Tokyo` → `Asia/Shanghai` 转换，**不得依赖含糊的手工字符串替换**；
5. 派生副本标记 `derivedForAnalysis: true` 或使用同等明确的命名和说明；
6. 派生副本**绝不导回 iPhone**，不与权威数据合并；
7. 不把原始或派生 JSON 提交仓库、发给 AI 或上传第三方。

若使用应用现有的时区平移导入能力，只能在隔离环境操作，并在导入前后核对抽样记录。**不得把「分析副本」冒充新权威状态。**

本轮不在正式 PWA 中新增「分析时区」功能，也不借此重写时间数据模型。

## 4. 冻结协议

### 4.1 允许的运行时改动

只有出现以下**阻断**时才允许提出修复：

- 数据丢失、损坏，或保存成功提示与真实状态不一致；
- 应用无法打开或无法进入记录流程；
- 无法创建、编辑、补录或撤销；
- 无法导出完整备份；
- 在隔离空环境中无法导入并恢复已有备份；
- 离线核心流程完全不可用。

任何修复仍需：复现证据、最小改动、回退方法、自动化测试和**维护者明确批准**。**发现问题不等于自动获得编码权限。**

### 4.2 明确不做

当前优先级新页面；AI 日复盘页面；新数据维度和完整分类体系；Agent 状态或独立 Agent 时间线；账号、后端、同步、WebDAV、自动备份；原生 App、Widget、Live Activity；支付、支持作者入口、商业化基础设施；周/月/年新增图表；游戏化、徽章、签到、提醒；英文版；图标、动画、视觉镀金；无证据的性能优化；**为作品集新增功能**。

### 4.3 新想法处理

冻结期间的新想法**只记录为候选**，不排期、不估时、不顺手实现。格式最多包含：触发场景、当前损失、临时绕法、出现次数、14 天后是否仍然重要。

技术性候选（非阻断缺陷、有复现证据、不含隐私数据）记入 [`docs/freeze-candidates.md`](freeze-candidates.md)；涉及真实活动内容的私人观察写在仓库外的每日副本里（模板见 `docs/dogfood-daily-template.md`）。**记录不等于排期**，也不构成修复授权。

## 5. Dogfooding 观察协议

### 5.1 有效记录日

满足以下即算有效：当天留下足以重建主要活动的时间线；不要求记录条数；不要求填满 24 小时；未确定或忘记的部分可诚实保持「未记录」；当天或次日完成一次 ≤ 2 分钟的检查。

目标：14 天中至少 **10 个有效记录日**。结合此前 18 个自然日的已有样本，累计真实记录可达到或超过 `CLAUDE.md` 的 28 天观察门槛；**但达到数字不自动解冻路线图。**

### 5.2 私有每日记录

仓库中**只保存空白模板**（`docs/dogfood-daily-template.md`），真实填写副本必须放在仓库外。不要求复制原始日志，不要求提交敏感活动内容。

### 5.3 14 天自用验收参考

以下是**个人使用门槛，不是市场验证**：

- 至少 10/14 个有效记录日；
- 日管理时间中位数 ≤ 5 分钟；
- 至少 8 个有效日可在次日重建主要活动；
- 没有不可解释的数据丢失或损坏；
- 空环境恢复演练通过；
- 完成 10 次高质量岗位投递；
- 发布 1 个作品集案例。

即使全部满足，也**只能说明它对维护者当前有用**，不能声称外部留存、付费或市场需求成立。

## 6. iPhone SE 冷启动取证

已知事实：v58 的 SW 提前注册和 `modulepreload` 在桌面模拟中改善了 waterfall，但 iPhone SE 真机**没有可感知改善**（见 `docs/postmortems.md` P33）。

冻结期只收集至少 **10 次真实冷启动观察**，模板见 `docs/dogfood-daily-template.md`。

边界：

- 不把桌面模拟结果当作 iOS 真机成功证据；
- 不为得到漂亮数字改变测试定义；
- 慢但可用时**先记录，不立即修**；
- 只有无法可靠打开或严重妨碍记录时才升级为阻断项；
- 下一次性能改动必须有可复现根因和前后真机对比，**禁止继续堆猜测式优化**。

> **版本探针注意**：v59（2026-07-15 15:44 部署）漏了版本联动，`src/ui.js` 的 `APP_VERSION` 仍是 `'58'`，因此**跑 v59 的设备在「更多」页显示「v58」**。v60 已修复此断裂。取证时若读到意外的版本号，先确认设备是否已收到 v60，再下「旧版本」结论。v58 冷启动复测记录于 07-15 11:49，早于 v59 部署，故当时「显示 v58 = 运行 v58」成立，**P33 的历史结论不受影响**。

## 7. 备份与空环境恢复演练

**目的**：验证「从空设备/空 origin 恢复当前完整备份」，不是验证历史版本回退。现有按 ID 安全合并导入 ≠ 用旧快照整体替换当前状态。

1. 在 iPhone SE 导出新的完整 JSON 备份；
2. 保留此前至少一份备份，**不覆盖唯一副本**；
3. **不清空 iPhone SE，不在权威数据源上演练**；
4. 使用空浏览器配置、备用设备或独立 origin；
5. 导入备份；
6. 核对记录总数；
7. 核对标签配置；
8. 抽查至少 3 个日期，含一个跨日或长时间段日期；
9. 核对导入后的时间和自然日是否符合该环境的时区语义；
10. 记录成功、失败和任何警告，**但不把真实内容写入仓库**。

若失败：停止试错，先保存失败前后的隔离环境状态和非敏感错误信息。**不得在权威设备上反复导入测试。**

## 8. 2026-07-30 阶段复盘

只允许形成三类结论：

1. **继续冻结并使用**：产品已满足当前需要，没有强证据支持新增功能；
2. **只修排名第一的真实摩擦**：该问题在 14 天内重复发生、造成明确损失，且最小修复可验证；
3. **停止主动开发**：保留现有工具，把时间转回求职和作品集。

**禁止因为积累了一批想法就整体解冻路线图。**

复盘时回答：有多少有效记录日；每日管理时间中位数；有多少天能在次日重建；最大摩擦实际出现多少次；冷启动观察结果；恢复演练是否通过；是否完成 10 次投递；是否发布作品集案例；下一项改动若不做会造成什么实际损失；**不继续开发是否反而更合理**。
