## D1 · 版本回退 / 快照（暂不做）

**评估日期**：2026-06-30 · **状态**：不落地

**现状真相**：
- 数据只有单一状态 `localStorage['timelog.v1']`，**没有历史/快照**。
- 导入是**按 id 合并**（`src/storage.js` `mergeImportedEntries`），**不是替换**。
- 因此**当前事实上没有「回退」能力**：导入旧备份只能并回旧条目，无法移除「旧备份里没有、但之后新增/删除过」的条目。把「导入旧备份」当回退用会得到错误结果。

**决策**：满 28 天真实记录前不做。回退/快照属于状态历史管理，紧邻被锁死的「可扩展/同步」边界，收益不抵复杂度与跑题风险。

**若将来真要做（不是现在）**：最小真回退原语＝「导入时提供『替换而非合并』选项」（一次性整体替换当前 `entries`），而**不是**引入多版本快照栈或云端历史。先有这个原语，配合现有「下载 JSON」就能手动回退，无需新数据模型。

## D2 · 付费转型 / 多人维护（暂不设计）

**评估日期**：2026-06-30 · **状态**：不落地

**疑问**：之后若转付费、且有其他开发者帮忙维护，该怎么处理？

**回答（付费与维护是两件独立的事）**：
- **付费**：时间尺是本地优先、零后端、无账号的静态 PWA，没有可计费的服务端。任何直接计费都会撞「无后端 / 无账号 / 无云同步」铁律。可设想的最小侵入路径是「赞助 / 一次性买断 + 可选云备份增值」，但这要等 v2 之后、且求职有实质进展再议。
- **多人维护**：靠**许可证 + `CONTRIBUTING` + 治理文档**约束，而不是改运行时结构。许可证已于 v31 落地（见下「许可证：已落地 AGPL-3.0」）；作为唯一著作权人保留商业双许可权，暂不需要 CLA。

**决策**：**许可证部分已落地**（AGPL-3.0，保护性、耐久、便宜）；**付费转型与云备份增值仍 gated**——现阶段（pre-28 天、求职未见实质推进）不设计、不落地，仅记录该决策本身。重申最大风险＝把「付费/治理规划」当成逃避面试推进的镀金。

### 许可证：已落地 AGPL-3.0（2026-07-01，v31）

**选择**：`AGPL-3.0-or-later`（非 MIT），配「作者保留商业双许可」姿态。`LICENSE` 全文 + 每个运行时文件 SPDX 头 + README「许可证与版权」节 + `CONTRIBUTING.md` 均已落地。

**理由**：
- **AGPL §13 堵住 SaaS 闭源克隆**：客户端 JS 本就完全可读，MIT 等宽松许可等于放任他人闭源套壳做成付费服务。AGPL 的网络条款要求「对外提供服务即须开源」，是对纯前端项目为数不多有牙齿的威慑。
- **唯一著作权人保留商业双许可**：现在只有一个作者，无外部贡献，可无 CLA 地在 AGPL 之外另行授权；这为将来「一次性买断 / 可选云备份增值」留了法律口子，又不必现在就设计付费。
- **法律威慑 + git 溯源即最优解**：前端代码无法技术性防拷贝，务实止损＝清晰许可（威慑）＋ 公开 git 历史（作者身份与时间线的可追溯证据）。

**未来可选项（备注，不做）**：软件著作权登记、商标注册——都属「求职有实质进展后再议」，此处仅登记备选，避免日后追溯。

## D3 · 推广策略（gated，暂不主动执行）

**评估日期**：2026-07-01 · **状态**：不落地（推迟主动推广）

**背景**：需求里含「写推广文案 / 选渠道」。主动社区推广正是 `CLAUDE.md`「别镀金」点名的 #1 风险——**用运营/推广动作逃避面试推进**的高发区。经评估重框为「求职证据工作」：本轮只做便宜、耐久、保护性的产出（许可证、博客草稿），**主动推广推迟**到博客产物就位或出现真实使用里程碑之后。

**决策**：现阶段不主动推广。先让「开发博客」作为可写进简历/面试的工程叙事产物落地；等有真实用户信号或求职实质推进后再谈渠道执行。

**渠道建议（就位后再用，不是现在）**：
- **中文社区先拿真实用户**：少数派 / 即刻 / V2EX / 掘金等，受众与「本地优先、隐私、零依赖」调性契合，且应用界面是中文，落地摩擦最小。
- **英文社区用「博客 dev-story」当内容源**：应用 UI 是中文，不适合直接向英文用户推应用；但**工程故事**（零依赖无构建 PWA、红线治理、postmortem 根因文化）可作为英文 dev-story 内容源投递 —— 正是「英文版当内容源、而非推应用」的设想。

**可核实的事实卖点（避免夸大，仅这两条已核对属实）**：
- **不主动保持屏幕常亮**：应用未使用 Wake Lock，不会强制屏幕常亮，符合「轻量、不打扰」定位。
- **完整备份是 Markdown 摘要 + JSON 导出**：摘要导出为 Markdown（可直接贴给 AI 复盘），完整备份为 JSON（复制/下载/分享），均在本地完成、不上传。

**重申**：以上属 gated 规划，登记以备日后，不构成本轮开工项。最大风险＝把「推广规划」当成逃避面试推进的镀金。

## D4 · 日视图交互调研（v36；提案 F 于 v45/v48 落地）

**评估日期**：2026-07-07 · **状态**：提案 E 仍为原型；提案 F 经用户授权于 v45 初版、v48 完整落地

**背景**：v34/v35「直接操纵 rail」（拖边界改时间）SE2 真机验收后被判定观感不及预期，v36 已回退为 v33 卡片列表（见 `CLAUDE.md` CHANGELOG v36、`docs/postmortems.md` P17）。用户要求研究是否有更优雅的日视图交互方向。为避免重蹈 P17——「demo 数据必须覆盖真实使用会撞到的极值分布，均匀数据会系统性藏起钳制类缺陷」——两个原型的 demo 数据统一采用真实分布：近 8h 整块睡眠、4~4.5h 专注工作块、外加一批 20~45min 碎片（洗漱/通勤/吃饭/刷手机），不用均匀数值。

**调研输入**：WebSearch 时间记录类 App 的日视图/列表交互模式（Toggl Track、Timery、ATracker 等）。确认 Toggl Track 官方模式为「Timer 屏幕的记录支持左滑删除/收藏」，是验证过的真实产品模式，而非臆造。

**曾产出两个可交互原型**（纯 demo 数据，非运行时资产；v53 已删除可执行副本，历史代码由 Git 保留）：

1. **`day-band.html`（提案 E · 比例日程带）**：日视图顶部加一条纯只读、不可拖的比例日程带——每段高度严格按分钟数比例分配，不做钳制、不做像素:分钟实时映射，因此结构上不可能重演 P17 的拖拽回弹问题。点一段只弹出摘要 popover（时间范围 + 内容 + 「编辑这条」跳转按钮），真正编辑仍下沉到下方卡片列表（与已发布运行时同款交互，零改动）。极薄碎片（视觉高度 <22px）自动隐藏文字、只留可点色带，避免挤爆。
2. **`day-swipe-cards.html`（提案 F · 滑动卡片）**：卡片右侧不再常驻编辑/删除图标按钮，改成左滑露出（Toggl Track 验证过的模式），正文因此更干净、留白更多；保留一个常驻「⋯」按钮兜底键盘/无触屏可达性——纯手势对键盘用户不可达，这是滑动模式必须补的口子。

**后续决策**：提案 E 仍未合入。提案 F 经用户明确授权后分两步落地：v45 先做左滑即编辑；v48 改为仅触摸/触控笔启用的 2×72px 编辑/删除轨道，补齐水平轴锁定、跟手、吸附、一次只开一张，以及纵向滚动/点空白关闭。原型里的常驻「⋯」兜底没有照搬，运行时用“点整卡编辑 + 编辑页删除 + 键盘 Enter/Space”提供完整非手势路径。

**剩余项**：提案 E 继续保持 gated，不因 F 已落地而自动开工。提案 F 的模型与双引擎自动化已完成，但 `touch-action`、惯性和纵横轴竞争仍必须在 Safari 浏览器与主屏 PWA 真机验收；手势类交互不能只靠 headless 宣布完成。

## D5 · 第三方产品审计意见的可行性分档（2026-07-08，gated）

**背景**：外部审计给出六个方向（原生 widget/灵动岛、半自动记录、界面冲击力、漏损预算、意义闭环、"三块空地"战略）。逐条对照本项目铁律（零运行时依赖静态 PWA、无后端/账号/云、数据只在 localStorage、v2 锁死）分档如下。**本条只登记判断，不构成开工项。**

### 档一 · 结构性做不了（web 平台无 API，需原生载体）

- **原生 widget / 灵动岛一键记录**：WidgetKit/ActivityKit 只对原生 App 开放，PWA 无任何入口。做它=转原生（Swift + App Store 分发）=产品形态转向，须另立决策，不是"加个功能"。
- **屏幕使用时间自动填充**：Apple 不向任何第三方（含原生 App）开放可导出的 Screen Time 原始数据（DeviceActivity 只给扩展内聚合报表），web 侧为零。**结构性不可行，与是否转原生无关。**
- **睡眠数据自动填充**：HealthKit 仅原生。web 不可行。

### 档二 · 可行但只有降级形态（28 天门后再议）

- **快捷指令近似"一键记录"**：iOS 快捷指令（可放主屏/锁屏 widget）打开 `?quick=switch` 之类带参 URL，`app.js` 启动时解析参数直接开表单。体验是"打开 PWA + 自动弹表单"，非真·一键，但零依赖、零后端。改动小，属工具打磨，门后做。
- **日历 .ics 本地导入（半自动消未记录）**：用户手动导出/分享 .ics，应用本地解析、按时间段提议成记录（复用 `mergeImportedEntries` 的确认式合并姿态）。无 OAuth、无 CalDAV、无云，隐私模型兼容。是"半自动记录"里唯一够得着的一块，但是真功能，门后做。

### 档三 · 纯本地可行、门后优先级最高（审计说的"结论力 + 意义闭环"恰好全在这档）

1. **异常天标记（T13）**：月/年汇总行按阈值（漏损%/未记录% 超标）标色/标记。纯 `stats.js` 层小改动，账本数据现成。**门后第一顺位**。
2. **漏损预算**：config 加一个"本周漏损 ≤N h"字段，ruler/摘要标注进度与超标。同样是统计层小改动。**门后第一轮可与异常天同批**。
3. **周回顾仪式 + 漏损账单**：`summarizeRange` 机制现成；加时薪 config 字段（localStorage，本地敏感数据可接受）即可算"本周刷手机 14h ≈ 时薪×14 机会成本"。**门后第二轮**。
4. **基于数据的下周建议**：内建 AI 违反零后端铁律，不做；但项目已有"摘要贴给 AI"闭环——便宜的增强是摘要导出里加周环比 delta，把闭环的原料喂得更好。规则式建议（如"漏损集中在 22:00 后"）可作为周回顾的一部分。

### 档四 · 观感/定位类（park）

- **人生格子图（LifeWidget 式）**：情绪冲击便宜是事实，但它不消费账本数据（纯出生日期算术），且审计自己也说"别跟 LifeWidget 比皮肤"。park，不排期。
- **ADHD 人群定位**：营销文案，归入 D3 推广 gate，随 D3 一起解冻。

### 对"三块空地"战略的修正

审计圈的三块空地里，**"记录自动化"这块对 web PWA 是半封死的**（widget/系统集成需原生；Screen Time 对谁都封死）——本项目真正够得着的是**结论力**（异常天、预算）和**意义闭环**（周回顾、账单、环比摘要），两者全是纯本地统计层改动，恰好也是四桶 taxonomy 这个核心资产的自然延伸。若某天判断"记录自动化"值得全力去占，那是「转原生 App」级别的产品决策，须另立条目评估，不在本仓库红线内。

**重申 v2 锁死**：以上全部 gated 在「28 天真实记录 + 求职实质进展」之后；本条审计意见本身就是最典型的镀金诱惑源——账本还没记满一个月，先别给账本装审计委员会。

## D8 · 发布存续、原生载体与 WebDAV 备份（gated，当前不实现）

**评估日期**：2026-07-14 · **状态**：记录边界与候选路径，不构成开工项

### 背景：用户信心其实包含三个不同问题

本条源于对“仓库转私有后 GitHub Pages 是否仍可使用”“PWA 缓存被系统回收后还能否启动”“是否需要原生 App 才能给用户长期信心”的连续讨论。三个问题不能混为一谈：

1. **源码可见性**：仓库是否公开，只控制 Git 历史、issue、文档和未发布文件的访问权限。已经发到网页端的 HTML/CSS/JS 必须交给浏览器执行，不能靠私有仓库隐藏；已经被 clone、下载或形成公开 fork 的副本也收不回来。
2. **运行时可获得性**：用户关闭页面后，是否还能取得启动 App 所需的静态文件。当前 PWA 靠 GitHub Pages 在线源站和 Service Worker 缓存共同保证；缓存是浏览器管理的 best-effort 存储，不是永久安装包。
3. **数据可恢复性**：设备丢失、卸载、清站点数据或浏览器回收存储后，记录能否恢复。原生安装包只改善运行文件的持久可获得性，**不会自动备份用户数据**；WebDAV/文件备份解决的是数据恢复，也不会让已经下线且缓存已失的 PWA 重新启动。

### 当前事实与承诺边界

- GitHub Free 个人/组织账号把发布中的仓库改为私有时，GitHub 会自动下线其 Pages；GitHub Pro/Team/Enterprise 可从私有仓库发布 Pages，但站点默认仍可向公网开放。私有仓库不等于私有网站。参见 [GitHub 仓库可见性说明](https://docs.github.com/en/repositories/managing-your-repositorys-settings-and-features/managing-repository-settings/setting-repository-visibility) 与 [GitHub Pages 建站说明](https://docs.github.com/en/pages/getting-started-with-github-pages/creating-a-github-pages-site)。
- 当前 `sw.js` 会预缓存全部运行时资产，并采用 cache-first 读取。已经访问或安装到主屏的设备在源站暂时不可用时通常仍能启动；但若 Cache Storage/Service Worker 被系统回收，且源站同时下线，刷新或重启后就不能重新取得 App。
- 页面仍开着时，已载入内存的代码可能暂时继续工作；这不构成关闭、刷新或重启后的可用性保证。
- 数据权威仍是当前 origin 下的 `localStorage['timelog.v1']`。仅运行缓存被回收时，记录可能仍在；若用户清除整个站点数据、卸载导致容器被清除、设备损坏或系统一并回收 origin 存储，记录也会消失。
- 当前可诚实承诺的是“本地优先、完整导出、稳定公开地址下可离线使用”，不能承诺“安装一次后，即使作者永久停服也一定能启动”或“本机数据永不丢失”。

### 候选路径 A：继续稳定托管 PWA（当前基线）

保持固定公开 URL 和静态源站，是成本最低且符合本项目身份的方案。用户清掉运行缓存后，只要源站仍在，浏览器就能重新下载 App；已有 `localStorage` 未被清除时，记录会重新出现。

这条路径的可信度来自：长期稳定的访问地址、清晰的本地数据说明、完整备份/导入能力、可验证的版本发布和停服前充分通知。它不要求原生 App，但也不提供脱离源站的永久启动保证。

仓库当前继续公开；若将来只想隐藏 Git 历史而保持公网 App，应先确认付费套餐和 Pages 设置，并认识到浏览器拿到的前端源码仍然可读。不要把“仓库转私有”当作代码保护方案。

### 候选路径 B：可独立启动的安装包（未来须另立项目）

若未来要作出更强承诺——“用户取得某个版本后，即使源站关闭，仍能从设备上的完整运行文件启动”——就需要把运行时资产随安装包交付。可以是：

- 用 Swift/Kotlin 重写的原生 App；
- 内嵌现有 HTML/CSS/JS 的原生壳；
- 桌面端打包载体或其它能把静态资源真正随安装包分发的方案。

不必为了离线启动保证重写全部 UI；关键是运行文件是否属于安装产物，而不是界面是否使用原生控件。但无论采用哪一种，签名、构建、商店/侧载、升级、平台审核和原生存储都会引入新的工程边界，直接违反本仓库“无构建 / 无运行时依赖 / 单页静态 PWA”的身份。因此必须放在**独立仓库/独立产品载体**中评估，不能把包装工具链塞进本仓库。

原生载体仍不能替代备份：卸载、清除 App 数据、设备丢失或换机仍会造成记录不可用。原生的额外价值是可进一步接入系统文件、后台任务、WidgetKit/ActivityKit 等平台能力；是否值得承担长期维护成本，要由真实用户需求证明。

### 候选路径 C：WebDAV 作为用户自有的备份通道（不是同步）

WebDAV 值得保留为门后候选，因为它可以让用户把备份写到自己选择的服务（例如支持 WebDAV 的网盘、NAS 或自建存储），不要求时间尺运营账号或自建业务后端。但若评估，第一版只能定义为**备份传输层**：

- 上传时间戳命名、不可变的完整 JSON 快照，避免静默覆盖唯一副本；
- 明确显示最近一次成功时间、文件名、大小与失败状态，网络失败不能伪装成已备份；
- 恢复必须由用户显式选择备份并预览，复用现有安全导入；若需要真正回到某一时点，应先实现 D1 所述“整体替换”原语，不能把按 ID 合并冒充恢复；
- 不做后台双向同步、不做最后写入者静默覆盖、不做多设备实时合并，也不引入时间尺账号；
- 默认保持端到端的本地隐私边界。若远端文件需要加密，应在客户端上传前使用 Web Crypto 加密，并先设计恢复密钥、忘记密钥和跨设备解密流程；“加密了但用户无法恢复”不算可靠备份。

纯浏览器 PWA 落地 WebDAV 有现实限制，评估时必须逐项验证：

- 远端必须使用 HTTPS，并正确允许浏览器 CORS；很多 WebDAV 服务默认不允许任意网页 origin 直接请求；
- 用户名、密码或 token 如何保存是安全问题，不能把长期明文凭据直接塞进 `localStorage`；
- iOS/PWA 后台执行不可靠，不能承诺关掉 App 后定时自动上传；
- endpoint、认证方式、`PROPFIND`/`PUT` 支持、配额、证书和代理兼容性差异很大，需要真实服务矩阵测试；
- 若用原生网络桥绕开 CORS/后台限制，就已进入候选路径 B 的独立原生载体，不再是本仓库的零依赖纯 Web 实现。

因此 WebDAV 的合理首版姿态应是“用户主动连接并手动备份/恢复”，不是“无感云同步”。在此之前，现有“存储备份”写入系统文件选择器（用户可自行选 iCloud Drive/其它文件提供方）仍是更简单、可理解、无需保存远端密码的基线。

### 解冻前必须回答的问题

即使跨过“28 天真实记录 + 求职有实质进展”的总 gate，也不能直接开工。须先完成一份小型设计评审，至少回答：

1. 要提升的具体承诺是哪一个：源站下线后可启动、换机可恢复、自动备份，还是多设备同步？
2. 有多少真实用户遇到过缓存回收或数据丢失，现有手动备份为什么不足？
3. 备份是追加快照还是覆盖单文件；保留多少版本；如何校验完整性；如何做一次真实恢复演练？
4. WebDAV 凭据、远端文件和本机数据各自的威胁模型是什么？忘记密码/密钥时如何诚实失败？
5. 浏览器/iOS 主屏 PWA/未来原生载体分别能保证什么，哪些能力只能 best-effort？
6. 独立安装包的发布、签名、更新和停止维护成本，是否真的小于“稳定托管 PWA + 文件/WebDAV 备份”的组合？

### 当前决策

- **现在不转原生、不接 WebDAV、不做同步，也不改变现有运行时。**本条仅把讨论和边界落盘。
- 当前继续以稳定公开 PWA + 完整手动备份作为基线；用户信心文案必须使用可兑现的承诺，不把 Service Worker 缓存说成永久安装。
- WebDAV 保留为未来“用户自有远端的手动/可验证备份”候选，优先级低于把现有导出—恢复流程验证扎实；它不得绕过 v2 对云端/跨设备同步的锁定。
- 若未来决定保证脱离源站仍可启动，另立原生/打包载体项目；若未来决定保证数据可恢复，单独设计备份与恢复。两项可以组合，但不能互相冒充。

重申最大风险：这类“长期产品化基础设施”很容易成为新的镀金方向。没有真实使用证据和求职实质进展时，正确动作是记录，而不是开工。
