# 1. 决策摘要

## 1.1 当前判断

【建议】**继续验证，但暂停继续扩建。**

【事实】当前产品已经具备日常记录、补录、区间编辑、切分、删除与撤销、跨日处理、天/周/月/年统计、JSON 备份、安全合并导入、PWA 离线和 Chromium/WebKit 自动化基础。

【事实】现有证据主要来自一名高频 Dogfooding 用户。它证明了产品可以服务这个人，也揭示了真实行为模式，但没有证明外部市场需求、留存、付费或增长。

【建议】下一阶段的唯一主问题是：

> 当一个高自主、低结构、经常被打断的知识工作者，用低摩擦时间线重建一天后，一份引用具体行为证据的“目标与实际行动偏差复盘”，能否让他在第二天真正改变一个动作？

若这个问题不成立，不应继续投入 Agent 集成、复杂分类、同步、英文版、App Store、支付、公开访问量或更多视觉打磨。

## 1.2 一句话产品定位

【建议】

> **Eigentime 是一款本地优先、可补录的行动时间线：它帮助被打断、并行工作且讨厌计时器的人重建真实一天，用具体行为证据判断当前目标是否被推进，并确定下一步该继续什么、停止什么。**

更短的传播表达：

> **不是计划下一天，而是从真实行动中纠偏下一天。**

## 1.3 最强差异化

【建议】核心选择：**目标与实际行为偏差分析**。

能力层级应当是：

| 层级 | 角色 |
|---|---|
| 数据基础 | 可编辑、可补录、允许模糊的真实行动时间线 |
| 核心价值 | 当前目标与实际行动的偏差审计 |
| 分析手段 | 有证据、可纠正、不过度评判的 AI/规则复盘 |
| 早期场景 | 人与 AI Agent 并行、等待、审核和返工 |
| 候选洞察 | 生产性逃避、维护性忙碌、无结果投入和切换成本 |

“AI”是方法，不是购买理由；“时间线”是证据层，不是最终结果；“Agent 状态”是早期高强度场景，不应立即扩张成 Agent 管理平台。

# 2. 证据等级：事实、推断、建议与待验证

## 2.1 已确认事实

【事实】私有 Dogfooding 样本包含 297 条记录，覆盖 18 个自然日，既有短时间段，也有长时间块、补录、跨日、混合活动和 AI 等待场景。

【事实】用户会记录明确任务，也会记录状态、意图、过渡阶段和“尚未决定做什么”。

【事实】补录和修改反复发生，因此补录不是异常路径。

【事实】日志中存在“开发同时看视频”“AI 在运行而本人转做别事”“等待额度或工具恢复”等并行场景。

【事实】设备记录时区为东京时间，但实际分析生活时区为 UTC+8；若不明确分析时区，会改变部分记录的自然日归属。

【事实】当前底层采用“点存储 + 区间 UX”：记录只存起点，结束由右邻记录、当前时间或日结算点推导。

【事实】当前桶归属依赖当前标签配置；修改标签桶可能追溯性改变历史统计解释。

【事实】当前导入是安全合并，不是精确状态恢复或历史回退。

【事实】当前产品没有账号、后端、遥测、可信用户数、留存或付费证据。

## 2.2 方向性推断

【推断】产品最可能服务的不是严格计时用户，而是需要“先留下不完整事实、之后逐步恢复”的人。

【推断】用户真正需要的不是知道“开发了几小时”，而是知道“为什么最重要的事情没有推进，以及明天要改变什么”。

【推断】Agent 工作流会强化等待、审核、返工和切换问题，但市场大小与长期频率仍未知。

【推断】四桶可以作为低认知负担的复盘框架，但具体名称和边界不一定适用于其他人。

【推断】现有高频使用可能受开发者身份、产品新鲜感、自我研究动机和持续调试驱动，不能外推为市场留存。

## 2.3 仍未验证的关键假设

【待验证】外部用户是否愿意连续主动记录至少 5 天。

【待验证】事后补录是否比计时器、日历、备忘录或自动追踪更适合目标人群。

【待验证】目标偏差复盘是否会产生第二天的真实行为变化。

【待验证】用户是否接受系统指出“维护性忙碌”“偏航”或疑似生产性逃避。

【待验证】AI/Agent 状态是否值得额外一次点击或字段。

【待验证】本地优先是否足以形成信任和付费理由。

【待验证】用户是否愿意为复盘价值作出真实金钱或时间承诺。

# 3. 目标用户与 Job to Be Done

## 3.1 第一目标用户

【建议】第一批测试者应是：

> **有明确现实目标、但一天高度自主、碎片化并经常被打断的知识工作者。**

优先招募：

- 使用多个 AI 工具的独立开发者或创作者；
- 自由职业者或远程知识工作者；
- 求职、职业转型或独立项目阶段的人；
- 一人公司或极小团队创始人。

筛选条件：

- 当前有一个有时间压力的目标；
- 工作、生活、信息消费和娱乐经常混在一起；
- 曾尝试并放弃计时器、日记、备忘录或任务系统；
- 常有“今天很忙，但最重要的事情没推进”的感受；
- 愿意每天花不超过 2–3 分钟复盘。

## 3.2 暂不优先

- 只需要客户计费和工时报表的人；
- 企业考勤、员工监控或审批场景；
- 只需要番茄钟的人；
- 不愿做任何主动记录的人；
- 核心需求是完整日历、待办或团队项目管理的人。

## 3.3 Job to Be Done

### 功能任务

【建议】

> 当一天被打断、混合和并行工作弄得难以回忆时，快速重建实际发生的事情，比较它们与当前目标的关系，并决定下一步。

### 情绪任务

【推断】

> 从“我今天是不是又浪费了”转为“我知道发生了什么，也知道下一步怎么改”，恢复掌控感，而不是增加内疚。

### 社会性任务

【待验证】

> 把不可见的独立工作整理成可向导师、合作方、客户或家人解释的结果摘要。

它可能有价值，但不是当前核心。

### 高频触发场景

1. 开始做事，但还没有决定精确任务；
2. 被消息、环境、工具或 AI 输出打断；
3. 忘记记录，需要事后补回一段；
4. AI 在后台运行，用户切换到别的活动；
5. 一天结束时感觉“很忙，但没有结果”；
6. 周计划失败，想知道时间实际去了哪里；
7. 当前目标与日常工具优化、信息浏览或维护工作发生冲突。

### 当前替代方案

- 记忆；
- 手机备忘录或聊天草稿；
- 日历；
- Screen Time；
- Toggl、aTimeLogger 等计时器；
- 自动桌面追踪；
- 日记；
- 把零散日志直接交给通用 AI；
- 完全不复盘。

# 4. 产品原则

## 4.1 记录原则

【建议】允许用户先记录模糊事实，不强迫先完成精确分类。

- 记录可以是任务、状态、意图、过渡或混合活动；
- 实时记录和事后补录同等重要；
- 低确定性应被标记和允许修正，而不是被禁止；
- 记录动作不应明显打断正在发生的事情；
- 记录目标不是“越完整越好”，而是足以支持后续判断。

目标体验：

- 打开到可输入尽量快；
- 一次普通记录目标 5 秒内完成；
- 外部验证中，单次记录中位数目标不超过 10 秒；
- 每日总管理时间目标不超过 5 分钟。

## 4.2 复盘原则

【建议】复盘必须导向一个动作，不能只增加图表和自我观察。

复盘输出应分为：

1. **观察到的事实**：引用具体记录或时间段；
2. **系统推断**：明确它只是推断；
3. **需要用户确认的问题**；
4. **一个下一步建议**；
5. **用户反馈**：接受、修改或拒绝。

禁止：

- 未经证据直接判断“你在逃避”；
- 心理诊断、道德评价或人格标签；
- 把相关性写成因果；
- 未经确认自动重分类或改写历史；
- 把 AI 输出直接写入权威数据。

推荐表达：

> 你把求职设为当前优先级，但今天有一段较长时间用于工具整理，且没有记录外部结果。你认为明天值得重复这段投入吗？

不推荐表达：

> 你今天一直在逃避求职。

## 4.3 成功指标原则

不要以这些作为产品成功：

- 用户每天记录更多条；
- 用户在应用中停留更久；
- 图表数量；
- 分类完整度；
- 页面访问量或公开下载数字。

更合理的指标：

- 用户坚持记录的有效天数；
- 单次记录和补录成本；
- 用户能否识别目标偏差；
- 复盘后是否作出明确决定；
- 第二天是否真的执行改变；
- 数据是否可靠、可导出和可恢复；
- 用户是否觉得被帮助而不是被评判。

# 5. 两周研究 MVP 边界

这不是从头重做应用。应基于现有 PWA，只增加验证核心价值所需的最小语义。

## 5.1 最多四个核心页面

| 页面 | 目的 | 最小内容 |
|---|---|---|
| 今日时间线 | 重建事实 | 快速记录、切换、补录、直接修改时间、跨日展示，不强迫先选项目 |
| 今日复盘 | 验证核心价值 | 当前目标投入、证据记录、宽泛长段、未确定区间、一个候选偏差、一个明日动作 |
| 当前优先级 | 提供分析上下文 | 只设置一个当前优先级、生效时间、什么算产生结果 |
| 数据与隐私 | 建立信任 | 分析时区、数据位置、导出、恢复、清除、AI 数据是否外发 |

不需要第五个页面。

## 5.2 MVP 中允许的最小结构化语义

记录时尽量只要求：

1. 发生了什么：自由文本；
2. 什么时候发生：可靠时间语义；
3. 与当前目标的关系：推进 / 维持 / 偏航 / 未确定；
4. 采集方式：实时 / 补录；
5. 确定性：精确 / 估计；
6. 可选 AI 状态：执行中 / 等待 / 待审核；
7. 结果状态只在复盘时确认，不在每次记录时强迫填写。

【建议】不要同时要求用户填写“活动领域、行为类型、目标关系、结果状态、注意力状态”五个维度。它们适合作为分析候选，不适合作为 MVP 表单。

## 5.3 AI 在 MVP 中的角色

第一轮不需要接入昂贵模型，也不需要建立自有 AI 后端。

优先顺序：

1. 本地生成结构化证据摘要；
2. 用确定性规则识别长段、宽泛描述、频繁切换和目标投入；
3. 研究阶段采用礼宾式复盘，或让用户使用自己已有的 AI；
4. 记录用户对结论的接受、修改和拒绝；
5. 只有证明复盘会改变行为后，才评估内建模型、BYOK 或本地模型。

# 6. 当前数据模型与分析能力的关键问题

以下是设计输入，不代表已经批准重构。

## 6.1 时间与时区语义不完整

【事实】当前核心时间是本地字符串起点；资料包同时存在设备记录时区、UTC、实际分析时区和自然日归属时区。

【建议】未来模型应能区分：

- 绝对时间；
- 记录时设备时区；
- 用户选择的分析时区；
- “一天”的归属时区。

跨天展示可以切片，但不能因此改变原始事实。

在没有完整迁移与回退设计前，不得直接改写历史时间。

## 6.2 当前桶配置会追溯重写历史解释

【事实】标签当前属于哪个桶，取决于今天的配置。

【风险】今天把某个标签从“维持”改成“主线”，过去统计也会变化，导致历史解释不稳定。

【建议】未来至少保存：

- 当时的当前目标版本；
- 记录或复盘时确认的目标关系；
- 分类来源和版本。

## 6.3 无法测量实时记录与补录成本

【事实】当前记录缺少明确的创建时间、最后修改时间、采集方式和确定性。

【建议】候选字段：

- `createdAt`；
- `updatedAt`；
- `captureMode: realtime | backfill`；
- `confidence: exact | estimated`。

这些字段首先用于研究和信任，不应增加表单负担。

## 6.4 点存储使相邻记录耦合

【事实】移动或插入一个起点，会改变前一段结束时间。

【建议】验证阶段继续保留点存储模型，不进行区间模型重写；所有编辑继续使用预览、最新数据复核、原子提交和撤销。

只有真实用户反复因相邻耦合受损，才评估底层迁移。

## 6.5 并行活动不应立即变成多条独立时间线

【事实】人类活动、媒体、AI 执行和等待可能同时发生。

【风险】若立即建立主活动、次活动、后台活动和 Agent 活动多条轨道，会增加录入负担，并产生合计超过 24 小时的解释问题。

【建议】MVP 只增加一个可选的“并行上下文”或 Agent 状态，不单独参加 24 小时总量计算。

## 6.6 缺少版本化目标

【事实】“求职是当前真正主线”等信息主要存在于自然语言和当前标签配置中。

【建议】目标对象至少需要：

- 名称；
- 生效时间；
- 结束或替换时间；
- 用户定义的有效结果。

第一版只允许一个当前最高优先级。

## 6.7 合并导入不等于恢复

【事实】当前导入按 ID 安全合并，无法精确回到某个历史状态。

【建议】在声称“可恢复”前，至少需要：

- 从空设备完成真实恢复演练；
- 恢复前自动导出当前状态；
- 明确区分“合并”和“整体替换”；
- 原子提交和失败回滚；
- 对旧版本备份的迁移测试。

完整事件溯源和历史版本浏览器可以后置。

# 7. 明确不做

两周研究和 MVP 阶段明确不做：

- 完整多维分类体系；
- 独立的第二条 Agent 时间线；

# 8. 两周外部验证计划

## 8.1 样本

先完成 12 次问题访谈，再邀请其中 8 人连续试用 7 天。

建议构成：

- 6 名使用多个 AI 工具的独立开发者或创作者；
- 3 名自由职业者或远程知识工作者；
- 3 名求职、转型或个人项目阶段的人。

至少一半应曾尝试并放弃计时器或时间记录工具。

## 8.2 给他们看什么

只展示：

1. 现有 PWA 的记录、补录和编辑；
2. 一条固定合成的混乱日时间线；
3. 一份引用证据的日复盘样例；
4. 一个可选的 AI 执行 / 等待标记；
5. 本地保存、导出和隐私说明。

不要展示完整路线图，不要用未来同步、Agent 自动集成或漂亮图表影响判断。

## 8.3 时间表

### 第 1–2 天：问题访谈

要求受访者回忆最近一个真实的混乱工作日：

- 当天最重要的目标是什么；
- 实际做了什么；
- 如何回忆和证明；
- 用过什么替代工具；
- 为什么停止使用；
- 最终有没有改变第二天行动。

### 第 3 天：冻结研究版本

只修复：

- 数据损坏或恢复阻断；
- 测试者无法完成记录、补录或复盘；
- 支持核心假设所必需的最小语义。

### 第 4 天：8 人入组

10–15 分钟 onboarding。每人现场完成：

- 创建一条实时记录；
- 补录一个缺口；
- 设置一个当前优先级；
- 完成一次简短复盘；
- 导出一次数据。

### 第 5–11 天：连续使用 7 天

收集：

| 指标 | 定义 |
|---|---|
| 有效使用日 | 当天形成足以支持复盘的时间线，不以记录条数越多越好 |
| 单次记录耗时 | 目标中位数不超过 10 秒 |
| 补录成功 | 能否独立补回一个缺口 |
| 日复盘完成率 | 是否阅读并确定一个明日动作 |
| 结论反馈 | 接受、修改、拒绝分别记录 |
| 行为改变 | 第二天是否实际执行了改变 |
| 管理负担 | 每日记录和复盘总时间，目标不超过 5 分钟 |
| 信任 | 是否理解数据位置、AI 边界和恢复方式 |
| 替代选择 | 哪一天更想使用日历、备忘录、自动追踪或不记录 |

默认不收集原始日志。测试者只提交匿名聚合数据和自己主动选择的脱敏片段。

### 第 12–13 天：回访与真实承诺

不要只问“愿不愿意付费”。提供一个真实选择：

- 小额支付继续使用 30 天；
- 或投入时间参加下一轮共创；
- 或明确停止。

口头说“以后可能会买”不算信号。

### 第 14 天：做方向决策

只允许三类结论：

1. 继续当前核心；
2. 收窄或转向；
3. 停止外部产品化，保留个人工具。

# 9. 继续、转向和停止标准

这些是小样本产品门槛，不是统计学证明。

## 9.1 继续目标偏差审计方向

大部分条件同时满足：

- 8 人中至少 5 人使用不少于 5 天；
- 至少 4 人能指出一个复盘导致的具体行动变化；
- 至少 3 人在第二天实际执行该变化；
- 单次记录中位数不超过 10 秒；
- 每日复盘不超过 3 分钟；
- 没有数据丢失或无法恢复事件；
- 至少 3 人作出真实付费或等价时间承诺；
- 用户普遍认为分析“有证据且可纠正”，而不是道德评价。

## 9.2 转向 A：时间线成立，AI 审计不成立

信号：

- 用户持续记录、补录和查看时间线；
- 但很少打开复盘，或认为复盘没有新增价值。

转向：

> 低摩擦、私密、可补录的生活行动时间线。

不要继续强推 AI。

## 9.3 转向 B：只有 AI 重度用户成立

信号：

- 普通知识工作者留存较差；
- Agent 重度用户明显更高频；
- 等待、审核、返工和上下文切换是主要价值。

收窄为：

> 面向 AI 时代独立开发者的人工注意力与 Agent 结果审计。

仍然不要变成 Agent 项目管理平台。

## 9.4 转向 C：复盘有价值，但主动记录负担过高

信号：

- 用户喜欢复盘结果；
- 但无法坚持主动记录；
- 希望利用日历或已有活动摘要恢复事实。

探索：

> 辅助重建，而不是完整自动追踪。

优先考虑用户主动导入并确认的数据，不先做后台监控。

## 9.5 停止外部产品化

出现多数信号时停止：

- 8 人中不足 3 人使用满 4 天；
- 没有人能举出复盘改变行为的具体例子；
- 用户普遍认为备忘录、日历或自动追踪已经足够；
- 超过三分之一认为分析冒犯、不可信或侵犯隐私；
- 用户只赞美界面和理念，却不愿继续使用或投入任何承诺；
- 为提高留存必须不断增加提醒、游戏化、自动化和复杂分类。

此时最诚实的结论是：

> 它是一款非常适合开发者本人的个人工具，但尚未被验证为外部产品。
