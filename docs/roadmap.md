## R1 · 性能 / 内存优化 —— 当前结论：基本无事可做

**现状测量（v26 · 2026-06-30，运行时资产）**：

| 部分 | 原始 | gzip |
|---|---|---|
| 代码（index.html + styles.css + 9 个 JS 模块 + sw + manifest） | ~166 KB | ~43 KB |
| 图标 PNG（含 icon-512 ~14 KB） | ~33 KB | ~30 KB |
| **整个 app 总计** | **~195 KB** | **~73 KB** |

- **运行时 RAM**：零依赖、零框架、零构建。JS 堆通常只有几 MB，且**绝大部分是 Safari / V8 自身开销**，不是 app 层能优化掉的部分。43 KB 的应用代码没有用户可感知的内存可省。
- **数据**：`localStorage['timelog.v1']`，每条 entry ≈ 124 B。28 天 ≈ 51–85 KB；满一年 ≈ 660 KB；均远低于 ~5 MB 上限。

**结论**：这已经是最轻量的那一档（零依赖静态 PWA）。**目前没有性能/内存问题可解**，主动优化＝典型镀金。

**真要做时（且只在确有可感知卡顿后，先测量再动手，按收益排序）**：
1. 图标瘦身（icon-512 PNG 优化或换更小尺寸），约省 ~10 KB —— 收益最小。
2. 年视图大量条目若实测掉帧，再考虑分段 / 虚拟滚动渲染。
3. 其余不动。

## R2 · 移动端「计时器挂件」降低记录阻力

**意图**：像锁屏计时器那样一眼可见、一点即记，进一步降低记录阻力。

**硬约束（必须先知道）**：
- iOS 的**锁屏 / 主屏 Widget（WidgetKit）**、**灵动岛 / Live Activity（截图里那种计时器）需要原生 App**（Swift / WidgetKit / ActivityKit）。**PWA / 网页做不到。**
- 上原生（或 Capacitor 套壳）会直接撞「单页静态 / 无运行时依赖 / 无后端 / 无框架」铁律 —— 等于换一个项目，不在本项目身份内。

**网页能做、且不破铁律的「降阻力」替代方案**：
1. **应用内「当前段」一键计时（收益最大、纯前端、零新依赖）**：打开即显示「正在做：X · 已 23min」，一键开始 / 结束当前段 —— 让 app 本身就是那个计时器。与现有续记 / 占位条模型天然契合。
2. **iOS 快捷指令（Shortcuts）入口**：用 Shortcut 打开 PWA 直达「新建记录」，可放主屏 / 轻点背面 / NFC 触发，逼近「挂件」手感，无需原生。
3. **Web Push 提醒（iOS 16.4+，需先加到主屏）**：定时「现在在做什么？」通知，点开直达记录。能力受限但可用。
4. 真要原生 Widget / Live Activity：单列为「是否做一个原生壳」的大决策，前提是放弃零依赖身份，门槛极高，默认不做。

**结论**：字面的「锁屏挂件」对 PWA 不可行；其**意图（降阻力）**可由方案 1 以最小代价实现 —— 但仍 gated，28 天 + 求职进展之后再议。
