## C1 · 点「更新应用」后无反应，`skipWaiting` 在 iOS 不可靠且无兜底

**触发场景**
2026-07-15 v60 → v61 真机确认。iPhone SE，Safari 与主屏 PWA 两个入口都出现：更新横幅正常出现、点击「更新应用」后版本纹丝不动。隐私模式（不注册 SW）直连网络显示 v61，证明服务端与网络无问题。

**第二次复现（2026-07-16，v61 → v62，主屏 PWA）**：完全退出 → 打开 app → 点「更新应用」→ 更多页仍显示 v61 → **再次完全退出**后重开才显示 v62。时序说明点击后 skipWaiting → controllerchange → reload 链路依旧没有可见效果，最终生效靠的是「所有客户端关闭后 waiting worker 自然激活」这条兜底路径，与首次复现口径一致。

**当前损失**
- 点了毫无反应、无任何反馈，用户合理地认为应用坏了或发布失败。
- 更严重的是**更新通道本身不可信**：冻结期一旦出现阻断级缺陷，修复也送不到设备上。这正是 P25 的复发形态（「每次真机测的都是旧缓存」）。
- 版本号读数曾因此连环失真（v59 探针把自己显示成 v58，v60 已修）。

**临时绕法（已验证可行）**
- PWA：从 App 切换器上滑**完全退出**，再打开。
- Safari：**关掉该站点全部标签页**（刷新无效），再重新打开。
- 原理：所有客户端关闭后，waiting worker 会**自然激活**，绕开 `skipWaiting` 这一环。
- 注意（2026-07-16 补）：打开后若又点了「更新应用」且版本未变，**再完全退出一次**即可；点击本身无害，只是没有可见效果。

**出现次数**
2（2026-07-15 Safari + PWA 均需绕法；2026-07-16 v61→v62 主屏 PWA 需二次完全退出，见上）。

**已知的代码侧事实（供复盘，不构成修复授权）**
- `src/app.js` `applyUpdate()`：`postMessage({type:'SKIP_WAITING'})` 发出后**没有任何超时兜底**；若 `controllerchange` 不触发就永久静默卡住。
- `applyUpdate()` 中 `reg.waiting` 为空时走 `window.location.reload()`——**这在 cache-first 下是空操作**，只会再喂一遍旧版。
- 本地真 SW 双引擎复现（WebKit + Chromium）：v58 → 发布 v61 → 横幅 → 点击 → 4 秒内成功更新。**干净路径工作正常**，故差异在设备/网络侧，不是逻辑写错。
- 现有 `npm run test:ui` 的更新用例**把 SW 整个 mock 掉**（只断言点击会 post `SKIP_WAITING`），真实的 skipWaiting → activate → controllerchange → reload 链路**无测试覆盖**。

**14 天后是否仍然重要**
待 2026-07-30 复盘回答。

## C2 · 预缓存 `addAll` 全有或全无，且失败完全无声

**触发场景**
`sw.js` install 走 `caches.open(CACHE).then(c => c.addAll(FILES))`，20 个运行时资产**任一**请求失败（网络抖动、CDN、VPN 干扰）即整体拒绝安装，设备静默停留旧版。

**当前损失**
- 失败**没有任何信号**：用户看不到、诊断里也没有。
- 症状与 C1 完全一致（「更新到不了」），但成因不同，**现场无法区分**——这会直接拖长下一次同类排查。

**临时绕法**
无。用户侧无从得知，只能靠再试一次。

**出现次数**
0。**2026-07-15 已排除为当日元凶**：线上 20 个文件逐个探活全部 200，且更新横幅出现本身就证明 install 成功（横幅只在 `reg.waiting` 存在时显示）。本条属于**尚未发生的风险**，不是已观测故障。

**重要边界**
「全有或全无」是**刻意设计**，不是缺陷：`CLAUDE.md` 红线「Service Worker 安装预缓存任一失败即拒绝安装」，理由是宁可让旧 worker 继续服务，也不要装一个半残缓存毁掉离线可用性。自测清单第 8 条同样要求「离线预缓存失败时旧 worker 继续服务」。
**若未来要动，需要动的是「无声」，不是「全有或全无」。** 把这条读成「应该允许部分缓存」是误读，会直接破掉离线红线。

**14 天后是否仍然重要**

## 复盘提示

C1 与 C2 症状相同、成因不同，且都指向同一件事：**更新通道的可达性与可观测性**。若 07-30 复盘选择「只修排名第一的真实摩擦」，二者应作为**同一个问题**一并评估，而不是拆成两个改动。

评估时先回答冻结协议要求的那个问题：**这项改动若不做，会造成什么实际损失？** 以及**不继续开发是否反而更合理？** ——绕法已验证可行且成本极低（完全退出重开一次），这是反对动手的有力论据；而「阻断级修复送不到设备」是支持动手的有力论据。两者都要摆上台面。
