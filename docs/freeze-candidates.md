## C1 · 点「更新应用」后无反应，`skipWaiting` 在 iOS 不可靠且无兜底

**触发场景**
2026-07-15 v60 → v61 真机确认。iPhone SE，Safari 与主屏 PWA 两个入口都出现：更新横幅正常出现、点击「更新应用」后版本纹丝不动。隐私模式（不注册 SW）直连网络显示 v61，证明服务端与网络无问题。

**第二次复现（2026-07-16，v61 → v62，主屏 PWA）**：完全退出 → 打开 app → 点「更新应用」→ 更多页仍显示 v61 → **再次完全退出**后重开才显示 v62。时序说明点击后 skipWaiting → controllerchange → reload 链路依旧没有可见效果，最终生效靠的是「所有客户端关闭后 waiting worker 自然激活」这条兜底路径，与首次复现口径一致。

**当前损失**
- 点了毫无反应、无任何反馈，用户合理地认为应用坏了或发布失败。
- 更严重的是**更新通道本身不可信**：冻结期一旦出现阻断级缺陷，修复也送不到设备上。这正是 P25 的复发形态（「每次真机测的都是旧缓存」）。
- 版本号读数曾因此连环失真（v59 探针把自己显示成 v58，v60 已修）。

**临时绕法（已验证可行）**
- PWA：从 App 切换器上滑**完全退出**，再打开。
- Safari：**关掉该站点全部标签页**（刷新无效），再重新打开。
- 原理：所有客户端关闭后，waiting worker 会**自然激活**，绕开 `skipWaiting` 这一环。
- 注意（2026-07-16 补）：打开后若又点了「更新应用」且版本未变，**再完全退出一次**即可；点击本身无害，只是没有可见效果。

**出现次数**
2（2026-07-15 Safari + PWA 均需绕法；2026-07-16 v61→v62 主屏 PWA 需二次完全退出，见上）。

**已知的代码侧事实（供复盘，不构成修复授权）**
- `src/app.js` `applyUpdate()`：`postMessage({type:'SKIP_WAITING'})` 发出后**没有任何超时兜底**；若 `controllerchange` 不触发就永久静默卡住。
- `applyUpdate()` 中 `reg.waiting` 为空时走 `window.location.reload()`——**这在 cache-first 下是空操作**，只会再喂一遍旧版。
- 本地真 SW 双引擎复现（WebKit + Chromium）：v58 → 发布 v61 → 横幅 → 点击 → 4 秒内成功更新。**干净路径工作正常**，故差异在设备/网络侧，不是逻辑写错。
- 现有 `npm run test:ui` 的更新用例**把 SW 整个 mock 掉**（只断言点击会 post `SKIP_WAITING`），真实的 skipWaiting → activate → controllerchange → reload 链路**无测试覆盖**。

**2026-07-17 部分处理（v64，维护者点名要求修）**
「无声装死」已修：`applyUpdate` 补 `statechange→activated` 第二成功路径 + 8 秒超时兜底，超时后横幅切换为「完全退出后重开」可执行指引。**未修也修不了的部分**：skipWaiting 本身在 iOS 的不可靠（OS 侧丢消息/不唤醒 waiting worker），本条继续开着观察后续真机表现；若 v64 后点击更新的成功率仍然为零，复盘时应评估是否干脆把按钮语义改为「引导完全退出」。

**2026-07-18 判读修正（v66→v67 与 v67→v68，诊断样本 + 维护者口述合并）**
初版判读曾写「无需完全退出的干净正例」，**维护者纠正：期间完全退出过一次**，且细看样本证据本就更微妙——两次升级形态完全一致：
- 22:54：navigate（v66 页、v66 缓存）→ **reload 样本＝旧代码 v66 + 只剩 v67 缓存**（自动 reload 确实触发、activation 已清旧缓存，但该次 reload 仍跑旧代码）→ navigate 才是 v67。
- 23:11：**reload 样本＝旧代码 v67 + 只剩 v68 缓存** → navigate 才是 v68。

**修正后口径**：v44 自动链路**可触发**（reload 样本是机器证据），但两次切版都落在随后的 navigate 上，与「完全退出→自然激活」一致——**自动链路能否独立完成切版仍未证实**，完全退出仍是当前唯一可靠路径。复盘按「链路可触发、端到端归因不明」评估；v68 起样本带 `SW态` 字段，下次升级窗口若出现 `w:installed` 悬挂即可定案。

**2026-07-20 定案观察（v68→v69，`SW态` 字段首次在真实交接时刻取到读数）**

```
17:39 navigate  v68 · 缓存 v68 · SW态 a:activated · 485ms
17:58 reload    v68 · 缓存已是 v69 · SW态 a:activated · 快照命中 · 135ms
      ← 维护者在此完全退出（2026-07-20 明确确认）
17:59 navigate  v69 · 缓存 v69 · 120ms
```

两个结论：

1. **C6 的「waiting 交接卡死」假说取得第一份直接反证**：交接时刻 `SW态` 是 `a:activated`，**没有 `w:installed` 悬挂**，也没有 installing 卡住。该字段正是为这一刻加的（v68），现在它答了——这条路不是卡死。
2. **自动链路仍未被证明能独立完成切版，且样本数升到 3/3**：三次升级（v66→67、v67→68、v68→69）形态分毫不差——自动 reload 触发了、activation 已清旧缓存（只剩新版一套），但**那次 reload 仍跑旧代码**，真正切版一律落在随后的 navigate 上；而本次维护者确认**期间完全退出过**。故 v64 之后「点更新→自己完成切版」依然零正例，**完全退出仍是唯一可靠路径**，横幅的「完全退出后重开」指引仍是用户实际脱困的那条路。

**问题定性因此可以收窄（复盘输入，不构成开工授权）**：故障不在「reload 不触发」（它触发了，3/3 机器证据），而在**「触发的那次 reload 没有拿到新版本」**。旧 `timelog-v*` 缓存此刻已被 activation 清掉，故 reload 拿到的旧代码不可能来自 SW 缓存——**强嫌疑是 WebKit 把主文档与模块图从进程内 HTTP/内存缓存直接交付、根本没问 SW**；这也解释了为什么「完全退出」有效（进程销毁即清掉这层）。若属实，则修法方向不是继续加固 `skipWaiting`（C1 原思路），而是**让 reload 发生在 `controllerchange` 之后、或给导航请求加绕过内存缓存的手段**。此假说的验证成本极低：下次升级窗口只需看 reload 样本是否仍是「旧代码 + 新缓存 + `a:activated`」三件套同时成立。

**14 天后是否仍然重要**

## C2 · 预缓存 `addAll` 全有或全无，且失败完全无声

**触发场景**
`sw.js` install 走 `caches.open(CACHE).then(c => c.addAll(FILES))`，20 个运行时资产**任一**请求失败（网络抖动、CDN、VPN 干扰）即整体拒绝安装，设备静默停留旧版。

**当前损失**
- 失败**没有任何信号**：用户看不到、诊断里也没有。
- 症状与 C1 完全一致（「更新到不了」），但成因不同，**现场无法区分**——这会直接拖长下一次同类排查。

**临时绕法**
无。用户侧无从得知，只能靠再试一次。

**出现次数**
仍为 0。**2026-07-15 已排除为当日元凶**：线上 20 个文件逐个探活全部 200，且更新横幅出现本身就证明 install 成功（横幅只在 `reg.waiting` 存在时显示）。
**2026-07-17 疑似现场经启动诊断判别后排除**：主屏 PWA「一直不提示升级」曾疑似本条，但用户回传的诊断样本显示 `timelog-v64(20文件,共2套)`——addAll 整批成功、install 完成，本条不成立。真实故障形态是安装完成后的 waiting 交接卡死，另立 **C6**。这次判别同时证明了启动诊断的 `cache`/`cacheCount` 字段确实终结了「现场无法区分」。

**重要边界**
「全有或全无」是**刻意设计**，不是缺陷：`CLAUDE.md` 红线「Service Worker 安装预缓存任一失败即拒绝安装」，理由是宁可让旧 worker 继续服务，也不要装一个半残缓存毁掉离线可用性。自测清单第 8 条同样要求「离线预缓存失败时旧 worker 继续服务」。
**若未来要动，需要动的是「无声」，不是「全有或全无」。** 把这条读成「应该允许部分缓存」是误读，会直接破掉离线红线。

## C7 · 过夜续记跳到今天后，昨天的 >3h 待确认段被遗落为「未记录」

**触发场景**
2026-07-18 用户报告。非 `longOk` 标签的过夜续记（「到今天硬终点」模式）写成昨/今两条后，保存即 `setSelectedDate(todayKey)` 跳到今天（`sheet_controller.js:1240-1242`，既定决策）。昨天部分（起点→24:00）若超 3h 即进入待确认，统计里计为**未记录**（`stats.js:81-87`）；而「确认」按钮只渲染在**那一天的日视图行内**（`ui.js:218`），周/月/年摘要只有不可点的「待确认 X」文字——不存在任何跨日召回机制。结果：用户以为记了，昨天统计里却悄悄多一块未记录。

**机制判定（代码侧事实，2026-07-18 审计）**
- 不是写入 bug：数据无损，昨天段完好、随时可确认。这是三个各自合理的既定设计（v9–v11 确认制 / v57 过夜续记 / 「记完跳今天」）的组合后果。
- 过夜是**唯一**「刚创建待确认段就导航离开它」的路径：普通 FAB 续记产生的 >3h 段留在当前视图、「待确认」标签当场可见；补录/编辑均停在所改的那天。
- 更深一层的模型问题：过夜表单里用户已显式断言两端（摘要明示「续昨晚 23:00 起 · 到今天 07:30」），保存这个动作在语义上就是确认；再要求翻回昨天补一次点击，属于确认制（本为拦截「疑似漏记拉长的段」）误伤「显式双端断言」。同理也适用于 v30 起的有界补录（起点+终点），它们同样落 pending。v17 曾在新建表单内联显示上一段确认块（该机制在后续重构中已消失），说明「确认应追到用户手边」有先例。

**可选修法（复盘时评估，不构成授权）**
- 方案 A（最小，模型层）：`planOvernightContinuation` 写入时随手落 `longConfirm`（昨点绑 startTs→midnightTs，今点绑 midnightTs→hardEndTs）；现有失效机制（相邻边界变化即删）原样保护。约 10 行 + smoke。
