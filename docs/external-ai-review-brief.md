# 外部 AI 评审上下文（v60）

> 快照日期：2026-07-15（v60 更新）
>
> 用途：帮助外部 AI 在准确理解项目后，提出下一步优化和设计建议。
>
> 权限边界：只做分析和建议，不修改代码、不创建 PR、不把建议写成已批准计划。最终是否实施由项目维护者另行评判。
>
> **当前阶段：2026-07-16 → 07-29 处于 14 天功能冻结**（见 [`docs/dogfood-freeze-handoff.md`](dogfood-freeze-handoff.md) 与 `decisions.md` D9）。冻结期只处理阻断级缺陷。**建议仍然欢迎，但默认不会在本阶段实施**，且新功能建议需等待 2026-07-30 阶段复盘。

## 1. 先给结论

时间尺是一个面向个人的求职主线时间记录与复盘工具。它不是自动计时器，也不是团队工时、习惯打卡或云端协作产品。它把一天按时间边界记录成连续区间，并用“主线 / 维持 / 漏损 / 未记录”四桶回答：时间实际去了哪里，哪些投入推进了当前目标。

当前发布版本是 **v60**（v59＝图标统一；v60＝版本仪式一致性修复，零业务逻辑）。记录—编辑—统计—备份的核心闭环已经完整，主要数据一致性和交互路径有 Chromium + WebKit 自动化守护；项目仍是个人、本地优先的静态 PWA，不应被描述为已有账号、服务器、遥测、跨设备同步或成熟商业化能力。

**当前阶段是 14 天功能冻结**（2026-07-16 → 07-29）。外部用户验证已延期——注意这**不代表验证通过或失败，而是尚未开始**：维护者当前没有候选用户，选择先把产品作为个人工具跑通闭环。因此**不得把本项目描述为「已验证市场需求」**；它当前只被证明对维护者个人有用。

当前最明确的未解问题是 **iOS 主屏 PWA 冷启动慢**。v58 的 Service Worker 提前注册和 `modulepreload` 在桌面模拟中改善了 waterfall，但用户于 2026-07-15 确认设备实际运行 v58 后，冷启动体感没有任何缓解。这个结果必须被当作评审输入，不能继续把桌面模拟改善外推成真机成功。（该复测记录于 07-15 11:49，早于 v59 部署于 15:44，故当时「更多页显示 v58」确实等于运行 v58，结论成立。v59 曾因漏做版本联动使该探针在 v59 设备上错显「v58」，已由 v60 修复。）

## 3. 当前产品与架构事实

### 3.1 运行形态

- 单页静态 PWA；GitHub Pages 从仓库根目录直接发布。
- 运行时只有 `index.html`、`styles.css`、9 个原生 ES modules、`sw.js`、manifest 和图标。
- 无构建、无框架、无 npm 运行时依赖；npm 仅用于 Playwright 开发期测试。
- Service Worker 预缓存全部运行资产，读取策略为 cache-first；缓存仍由浏览器/系统管理，不是永久安装包。
- 数据不上传。没有账号、后端、遥测、远程日志或可靠的全局用户数。

### 3.2 数据模型

- 权威记录：`localStorage['timelog.v1']`。
- 标签配置：`localStorage['timelog.config']`。
- 每条 entry 只存起点 `ts`；结束时间由右邻记录、当前时间或自然日结算点派生，属于“点存储 + 区间 UX”。
- 本地自然日 `00:00` 是统计硬边界；周/月/年由每日独立统计累加。
- 四桶为 `job / maintain / leak / unrecorded`；计划 `pending` 单独呈现，不混入四桶完成统计。
- 区间编辑、切分、删除和过夜写入先由事务 planner 计算预览与结果签名，提交前在同一次最新 `load()` 对象图上重算，防止跨标签或导入导致静默覆盖。

### 3.3 已完成的核心能力

- 日视图连续日志：开始—结束区间编辑、补录、切分、删除确认和 8 秒撤销。
- 今天尾段支持“至今”或固定结束；计划记录有本地自然日窗口和既有时间未变豁免。
- 昨日尾部未记录可显式选择跨午夜续到今天，或只记到 `24:00`。
- 天/周/月/年统计和下钻；长段确认避免错误分类污染统计。
- 完整 JSON 备份、系统文件存储/分享、安全合并导入、逐条冲突处理和时区平移建议。
- 当前视图 Markdown 摘要可交给 AI 复盘，但应用本身不内建 AI。
- 本地多标签页变更检测、PWA 更新提示、Safari 启动快照和回前台分钟刷新。

## 4. 已验证优势与不能夸大的地方

### 可以据代码和测试主张

- 写路径重视最新数据复核和原子结果，已有多轮真实 bug 反推出来的守卫。
- 导入不会把冲突静默塞进本机数据；用户必须逐条选择后才原子提交。
- UI 同时覆盖 Chromium 和 WebKit；测试包括响应式、区间事务、导入、分享降级、跨标签、PWA 更新和压力场景。
- 运行时轻量，模块 parse/execute 不是当前 3.8s 级冷启动的主要占比。

### 不能据此主张

- “添加到主屏后永久可用”：系统可以回收 Service Worker/Cache Storage，源站下线后可能无法重新取得 App。
- “备份可精确恢复到某个历史状态”：当前导入是安全合并，不是整体替换或版本回退。
- “数据永不丢失”：清除站点数据、设备损坏或未做外部备份都会丢失本地记录。
- “v58 已解决冷启动”：真机已确认没有可感知改善。
- “已有大量用户验证”：项目没有遥测或可信用户计数；公开文档也没有证明 28 天 gate 已满足。
- “自动化完全零 flaky”：2026-07-15 完整 UI 门禁最终以退出码 0 完成（208 passed，2 个 WebKit 用例首跑失败后重试通过）；近期 WebKit 的滚轮/区间等时序敏感用例仍有偶发重试信号，不能把“最终全绿”包装成测试基础设施完全稳定。

## 5. 当前首要未解问题：iOS 主屏 PWA 冷启动

已知证据：

- 现场曾观察到约 3.8s 冷启动，体验经历白屏/骨架到内容。
- 桌面 Chromium 测量表明，完美缓存命中时即使 CPU 10× 节流，模块执行仍远低于该量级；慢主要出现在网络等待或缓存未命中。
- v58 将 Service Worker 注册提前到 `<head>`，并对 9 个模块加入 `modulepreload`；桌面 6×CPU + slow-3G 模拟从约 4463ms 降到 3675ms。
- 2026-07-15 用户在“更多”页确认设备运行 v58 后复测，主观上“一点没有缓解”。没有新的同条件精确计时，因此只能判定“无可感知收益”，不能声称数值完全不变。

仍未知：

- 慢启动时页面是否已经被目标 Service Worker 控制。
- Cache Storage 是否存在、是否命中、是否被 iOS 定期或按存储压力回收。
- 慢发生在导航响应、模块获取、模块执行、快照恢复还是 WebKit 进程/主屏容器启动。
- 浏览器 Safari 标签页与主屏 standalone 是否共享同一套有效缓存生命周期。

评审必须针对这些未知提出**最小、可在真机取得证据的 instrumentation**，而不是直接再给一批性能技巧。若认为 PWA 平台约束使问题不可解，应明确说明证据门槛，并比较启动屏、资源合并/构建、原生壳或真正原生 App 等破界方案的实际代价。

## 6. 已知设计约束与历史教训

- **P1**：先 `find` 后 `save` 必须共用同一次 `load()` 对象图；否则会改一张图、保存另一张图。
- **P17**：均匀 demo 会藏起真实极值分布下的交互结构缺陷；原型必须覆盖长睡眠、长专注和碎片段混合。
- **P23**：iOS `visualViewport` getter 和事件时序不可靠；不要再建依赖精准键盘时序的状态机。
- **P25**：部署新版不等于用户已经加载新版；Service Worker 更新可达性必须单独验证。
- **P29**：启动快照能消除空壳，但首轮重建会把闪烁换一种形式重新带回来。
- **P33**：桌面节流是定位工具，不是真机 PWA 成功证据；合理猜测必须和已证实根因分开。

对已经讨论过的方向，若要重新提出，必须说明新增证据或不同推理：

- 框架、打包器、模块合并和懒加载；
- 原生壳、Swift/Kotlin、Widget/Live Activity；
- WebDAV、文件提供方、备份、恢复和跨设备同步；
- 自动读取 Screen Time/HealthKit/日历；
- 付费、支持作者、用户计数和推广；
- 更多统计、图表、里程碑与设置扩张。

## 7. 铁律可以挑战，但必须明码标价

当前默认铁律是：无运行时依赖、无构建、原生 ES modules、无账号/后端/云同步、本地优先、隐私数据不出设备；新功能还受“累计 28 天真实记录 + 求职有实质进展”gate 约束。

本次外部评审**允许挑战全部铁律**。但建议必须分成两类：

- **A · 边界内优化**：保持当前 PWA 身份和数据边界，可在现有仓库实施。
- **B · 架构转向候选**：需要打包器、运行时依赖、原生载体、后台服务、账号、云存储或其它破界条件。必须列出被打破的规则、为什么值得、迁移/回退路径、长期维护成本和启动该方向所需的真实证据。

“技术上能做”不是充分理由。若 B 类建议只是把复杂度搬到另一个层，没有解决已证实问题，应明确判为不值得。

## 8. 建议评审优先级

按以下顺序评价下一步：

1. 数据正确性、可恢复性和用户信任；
2. 降低记录阻力、提高每日结论价值；
3. PWA 启动、离线和真机可靠性；
4. 视觉提升与交互细节；
5. 增长、商业化、原生化等产品扩张。

这不是禁止低位方向，而是要求高位风险先被看见。任何视觉或商业建议若引入数据风险、隐私退化或维护负担，必须显式扣分。

## 9. 隐私与证据边界

- 只能使用仓库代码、固定 demo 截图、自动化结果和不含个人线索的现象摘要。
- 不得要求上传真实 `localStorage`、真实备份、真实记录截图、公司/面试信息或其它个人数据。
- 不知道的事实必须写“未知，需要验证”，不能虚构用户研究、留存、活跃、转化或市场规模。
- GitHub Stars、访问量和 Release 下载量不能冒充真实使用人数。

## 10. 外部 AI 的固定输出要求

请只做评审，不直接修改仓库。输出应包含：

### 10.1 理解校验

- 用不超过 10 条复述项目目标、架构和当前状态。
- 指出文档与代码之间任何疑似矛盾；无法验证时标为未知。

### 10.2 阻断项

- 先列数据损坏、安全、隐私、恢复或发布存续方面的阻断问题。
- 若没有发现，明确写“未发现”，不要为了显得全面而编造。

### 10.3 最多 5 项排序建议

每项必须包含：

1. 解决的具体问题与现有证据；
2. 建议方案；
3. A 类边界内优化或 B 类架构转向；
4. 用户收益；
5. 复杂度和长期维护成本；
6. 数据、隐私、兼容性和回归风险；
7. 预计影响的模块/文件；
8. 最小验证实验；
9. 成功标准与停止条件；
10. 为什么排在这个位置。

### 10.4 冷启动专项

- 按置信度排序可能根因，严格区分证据与推断。
- 给出下一次真机 instrumentation 的最小设计、采集字段和隐私边界。
- 说明哪些建议在 Cache Storage 被回收时仍有效，哪些只在缓存存在时有效。
- 比较继续 PWA、加入启动屏、引入构建/资源合并、原生壳和原生重写，但不要未经证据直接宣布转原生。

### 10.5 明确不建议做什么

- 列出当前不值得做的方向及原因，尤其是高复杂度、低证据或重复历史失败的建议。

评审输出只是候选输入。维护者会逐项质询、选择或否决；在明确批准前，不进入实现、版本仪式、提交或发布。
